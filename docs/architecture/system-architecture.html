System & Software Architecture

Overview

System Topology

One public entry point, fourteen independently deployable services each with their own database, and one event bus tying them together asynchronously. Nothing below the gateway is reachable from the internet directly.

Redis, Elasticsearch, object storage and the observability stack are shared infrastructure — each service opts into exactly what its own schema documents (see the Redis/Kafka notes per service in the Data Dossier), not a blanket dependency every service carries.

Application Architecture

Hexagonal Architecture, Per Service

Every one of the 14 services follows the same internal shape: a domain core that imports nothing framework-specific, wrapped by ports (interfaces) that adapters plug into from both sides. Worked example below using Feed & Content — the same layout applies whether the service is a simple CRUD store like Elections or a saga like Membership & Affiliation.

The domain core (Post, Visibility, PostPublishedEvent) has zero Spring/JPA/Kafka imports — it compiles and unit-tests with nothing but the JDK. Everything framework-specific lives in an adapter, which is exactly why swapping Postgres for a different store, or REST for GraphQL, never touches business logic.
